## 根本原因分析

**位置**:`src/Scalance.Protocols/Ssh/SshClient.cs:81-86`

```csharp
public async Task<string> RunAsync(string command, CancellationToken ct = default)
{
    using var cmd = _client.CreateCommand(command);
    cmd.CommandTimeout = TimeSpan.FromSeconds(5);  // ← 寫死 5 秒
    return await Task.Run(() => cmd.Execute(), ct);
}
```

兩層問題:

1. **CommandTimeout 寫死 5 秒太短**。對 SCALANCE 這種輸出可能有 ANSI 控制碼、可能會分頁的命令,5 秒不夠。
2. **更根本的是用 exec channel 而不是 interactive shell**。`Renci.SshNet.SshClient.CreateCommand(...).Execute()` 開的是 SSH `exec` channel,執行單一指令、靠 channel EOF 當結束訊號。SCALANCE 的 SSH server 看起來不會在 `show …`、`ping …` 這類指令完成後送 EOF — 它假設使用者在互動 shell 裡、用 prompt 偵測指令邊界。`cmd.Execute()` 因此會卡在等 EOF,直到 CommandTimeout 觸發。

**佐證**:
- VLAN 走 SNMP — ✅ 通
- 子網/IP 走 SSH(逾時)→ SNMP fallback — 有資料但少欄位
- NTP / Ping 純 SSH — ❌ 全部 5 秒逾時
- 不是網路或認證問題:同一個 SSH 設定,SNMP 那條走 UDP 161 的路徑完全正常;TCP/22 也通(`測試連線` 那邊有做 1.5 秒 TcpPing 檢查並通過)

## 建議修正(依優先序)

### 1. 立即:把 SSH 通道從 exec 改成 ShellStream(必要)

`SshClient.RunAsync` 改用 `client.CreateShellStream(...)`,送一行指令、讀到下一個 prompt(`cli#` / `cli(config)#` / `cli(config-…)#`)為止。這樣對:
- SCALANCE 的「沒 EOF」行為相容
- 跨指令模式狀態可以保留(`RunBatchAsync` 已經是這個假設)
- 可以順便處理 `--More--` 分頁(送空白鍵或 `terminal length 0` 一次)

### 2. 立即:CommandTimeout 改可調 + 預設拉長(防呆)

即使改 ShellStream,給 caller 傳入 timeout 比較安全。建議:讀類 10-15 秒、寫批次 30-60 秒、`ping` 看 count×timeout 算。

### 3. 短期:把 publish/ 重 build 一次

`publish/win-x64/` 是 2026-04-16 的舊產物,連 `show ntp info` 修正都還沒進去。重 build 後 NTP 命令名才會正確(雖然 SSH 通道沒改之前還是會逾時,但至少把舊 bug 從 baseline 掃乾淨)。

### 4. 短期:NTP write 的 forceExecute=true 要再三斟酌

`ScalanceCliDriverBase.RunOrPlanAsync` 對 NTP 有 `forceExecute=true`(略過 DryRun)。SSH 既然現在會 timeout,寫入會失敗;但如果之後修好 SSH,使用者第一次按「套用」會直接寫入真機而沒有預覽機會。建議改成 NTP 也走 DryRun,同時在 UI 上讓使用者選「預覽 / 直接寫入」。

### 5. 中期:diagnostic 失敗訊息改更具體

現在 UI 看到的訊息是 `Command 'show ntp info' timed out.`,使用者很難判斷是不是該換 IP / 換密碼 / 重啟。建議在 timeout 失敗時加一句「請檢查連線狀態頁的 SSH 路徑診斷」並把實際 SSH 連線狀態(已連 / 已認證 / 卡在哪一步)記到操作記錄。
